# 体验评测 — 开源仓库出题指南（外包同学版）

> 版本：v2.0 | 日期：2026-04-15

---

## 一句话说清楚你要做什么

**选一个开源仓库，给它加一个功能（或修一个 bug），用 AI Coding 工具实现，然后用打包 prompt 把题目打包提交。**

---

## 核心流程（四步）

### 1. 选题

仓库里的 `candidate_tasks_all.json` 有一批候选题，你可以从里面挑，也可以**自己出题**。

**候选题只是参考**，不是必须做的。如果你在分配给你的仓库里发现了更好的出题点（比如某个 feature request、某个 TODO、某个明显可以加的功能），完全可以自己出。

**自己出题的要求：**
- **必须在下面"人员分工"里分配给你的仓库中选**，不要跑到其他仓库出题
- 锁定一个 base commit（`git log --oneline -1` 记下 hash）
- 想一个合理的功能需求或 bug 修复场景

### 2. 用 AI 工具实现

1. Clone 仓库，checkout 到 base commit
2. 搭好开发环境（按 README 跑）
3. 打开你的 AI Coding 工具（Cursor / Windsurf / Cline / Claude Code / Copilot Chat 都行），把需求丢给 AI
4. AI 生成代码后验收，不通过就反馈给 AI 继续改，直到功能完成
5. 跑一遍回归测试确认没破坏已有功能

### 3. 打包（重点）

功能做完后，把仓库里的 **`ISSUE_PACKAGING_PROMPT_EXTERNAL.md`** 的完整内容作为 prompt 发给 AI，让它帮你自动打包。

AI 会帮你生成：
- `ISSUE.md`（题目描述 + 验收标准）
- `init/`（改动前的代码快照）
- `final/`（改动后的代码快照）
- `reproduce.sh`、`environment.yml` 等环境文件

**AI 打包完后，你必须逐项人工检查以下内容：**

**一、检查 ISSUE.md 内容**
1. **需求描述是不是口语化**：像真实用户在跟 AI 聊天，不是产品文档（见下面"题面要求"）
2. **需求是否完整准确**：对照你实际实现的功能，ISSUE.md 里写的需求是不是都覆盖到了？有没有漏掉的功能点？有没有 AI 瞎编的需求？
3. **验收步骤是否具体可操作**：每一条都必须是"运行什么命令 → 看哪里 → 应该看到什么"，不能只写"功能正常"这种模糊描述
4. **验收步骤能不能跑通**：实际按步骤跑一遍——init 状态下功能不存在，final 状态下功能正常

**二、检查目录结构**
1. **`init/` 是不是 base commit 的干净状态**：不应该包含你的改动
2. **`final/` 是不是实现功能后的状态**：应该包含你的所有改动
3. **`changes.diff` 是否存在**且只包含功能相关的改动（没有无关文件、格式化噪音）
4. **`reproduce.sh` 和 `environment.yml` 是否存在**
5. **没有 `workspace/` 目录**（这个不应该提交，由 reproduce.sh 自动生成）
6. **没有 `node_modules/`、`__pycache__/`、`.git/` 等不该提交的目录**

### 4. 提交

AI 打包完会帮你推到 `https://github.com/Qinyun718/external_manual_dataset.git` 的独立分支。一题一分支，分支名格式：`<你的名字拼音>/<task-slug>`。

## 题面要求（很重要）

ISSUE.md 里的「用户请求」部分必须**口语化**，像一个真实用户在跟 AI 工具聊天时说的话，而不是一份正式的技术文档。

**好的写法：**

> 我在用这个项目，发现它没有 Stepper 组件，能帮我加一个吗？需要支持横向和纵向两种布局，每一步可以点击跳转，当前步骤要有高亮样式。对了，最好能跟项目已有的主题系统对接，不要写死颜色。

**不好的写法：**

> 需求：实现 Stepper 组件。功能要求：1. 支持 horizontal/vertical 两种 orientation prop。2. 支持 clickable 步骤导航。3. 当前步骤高亮。4. 集成主题系统。

区别在于：好的写法像人在说话，有语气、有上下文、有"对了"这种口语转折；不好的写法像产品文档，列需求点。

**如果是多轮对话场景（用户先提了一个需求，看到结果后又追加了新要求），要还原这个过程**，不要把所有需求压成一段话。

## 质量红线

### 难度要求

| 你和 AI 的交互轮数 | 判断 | 处理 |
|---------|------|---------|
| 1-3 轮就通过 | 太简单，没区分度 | 弃题或加难度 |
| 4-7 轮 | 偏简单 | 可以保留，日报里说明 |
| 8-15 轮 | 合适 | 正常提交 |
| 15+ 轮仍未完成 | 可能太难 | 超 3 小时弃题 |

### 什么是好题目

- 需要 AI **读懂已有代码**才能做对，不是从零写一个独立的新东西
- 需要改 3 个以上已有文件
- AI 首次实现大概率有 bug，需要你反馈后才能修好

### 什么是差题目

- 新增一个和现有代码几乎没交互的独立文件
- AI 一次就全部通过
- 只改 1 个文件就搞定

## 常见问题

**环境搭不起来？** → 检查 README 前置要求（Node/Go/JDK 版本），30 分钟搞不定就弃题换题。

**AI 代码跑不通？** → 把完整报错反馈给 AI。多轮不行就换提问方式（拆小需求、先做第一步）。

**验收步骤不清楚？** → 群里 @ 负责人，不要自己猜。

**回归测试挂了？** → 看是你的改动导致的还是 base commit 本来就挂的。前者让 AI 修，后者在日报备注。

## 重要提醒

1. **每道题必须基于指定的 base commit**，不要用最新代码
2. **候选题是参考，鼓励自己出题**——但必须在分配给你的仓库范围内选
3. **打包全靠 AI，但人工检查不能省**——ISSUE.md 内容、目录结构、验收步骤都要逐项过一遍
4. **弃题不丢人**，比卡 2 小时强
5. **不确定就问**，不要自己猜
